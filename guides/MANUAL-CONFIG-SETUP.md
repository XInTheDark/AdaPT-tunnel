# Manual config reference

This guide is for operators who want the lower-level raw config reference rather than the guided CLI flow.

The supported client distribution artifact is still the single-file `.aptbundle` generated by `apt-edge add-client`.
The client example in this guide describes the **logical config embedded inside that bundle**, not a file that `apt-client up` consumes directly anymore.

If you just want the easiest supported path, use:

- `guides/DEPLOYMENT.md`

## 1. Generate raw server keys

```bash
./target/release/apt-edge gen-keys --out-dir /etc/adapt
```

This produces:

- `/etc/adapt/shared-admission.key`
- `/etc/adapt/server-static-private.key`
- `/etc/adapt/server-static-public.key`
- `/etc/adapt/cookie.key`
- `/etc/adapt/ticket.key`

## 2. Generate a standalone client identity (advanced/server-side use)

```bash
./target/release/apt-client gen-identity --out-dir /etc/adapt
```

This produces:

- `/etc/adapt/client-static-private.key`
- `/etc/adapt/client-static-public.key`

Copy the client public key to the server.

## 3. Prepare the config files manually

Use these templates:

- `guides/examples/server.toml`
- `guides/examples/client.toml` (reference for the config embedded inside a client bundle)

Key points:

### Server config

- `bind` ‚Äî UDP listen address
- `public_endpoint` ‚Äî what clients should dial
- `d2_bind` ‚Äî optional UDP listen address for the `D2` QUIC-datagram carrier
- `d2_public_endpoint` ‚Äî optional client-facing `D2` endpoint, usually `host:443`
- `d2_certificate` ‚Äî optional server certificate for `D2`; supports `file:/path`, inline PEM, or base64-encoded DER
- `d2_private_key` ‚Äî optional server private key for `D2`; supports `file:/path`, inline PEM, or base64-encoded DER/key data
- `stream_bind` ‚Äî optional TCP listen address for the `S1` fallback carrier
- `stream_public_endpoint` ‚Äî optional client-facing `S1` endpoint, usually `host:443`
- `stream_decoy_surface` ‚Äî whether invalid unauthenticated stream input gets a decoy-like HTTP surface
- `mode` ‚Äî numeric runtime mode (`0` = speed, `50` = balanced, `100` = stealth)
- `endpoint_id` ‚Äî deployment identifier
- the core 32-byte APT keys support either inline hex or `file:/path`
- `tunnel_local_ipv4` + `tunnel_netmask` define the tunnel subnet
- `tunnel_local_ipv6` + `tunnel_ipv6_prefix_len` optionally enable an IPv6 tunnel subnet, but only turn these on when the host Linux IPv6 stack is enabled
- `enable_ipv4_forwarding` / `nat_ipv4` control Linux IPv4 forwarding and NAT
- `enable_ipv6_forwarding` / `nat_ipv6` control Linux IPv6 forwarding and NAT66
- `push_routes` usually contains `0.0.0.0/0` for full tunnel; add `::/0` too if you enabled tunnel IPv6
- `push_dns` is applied automatically where supported by the client platform
- `allow_session_migration` ‚Äî enables authenticated path revalidation / migration handling
- `[[peers]]` must include each authorized client public key and assigned tunnel IPv4, and can also assign `tunnel_ipv6`
- for the recommended per-user model, each `[[peers]]` entry should also set:
  - `auth_profile = "per-user"`
  - `user_id = "..."` (or let it default to `name`)
  - `admission_key = "file:/path/to/user-specific-admission.key"`
- for the older shared-deployment model, leave `auth_profile` at its default (`shared-deployment`) and do not set a peer-specific `admission_key`

### Client config

- `server_addr` ‚Äî server host:port the client should connect to
- `enable_d2_fallback` ‚Äî enables conservative `D1 -> D2` fallback when `d2_server_addr` and `d2_server_certificate` are present
- `d2_server_addr` ‚Äî optional `D2` QUIC endpoint, usually `host:443`
- `d2_server_certificate` ‚Äî pinned `D2` server certificate; supports `file:/path`, inline PEM, or base64-encoded DER
- `stream_server_addr` ‚Äî optional TCP `S1` endpoint for fallback, usually `host:443`
- `mode` ‚Äî numeric runtime mode (`0` = speed, `50` = balanced, `100` = stealth)
- `preferred_carrier` ‚Äî `d1`, `d2`, `s1`, or `auto`
- `endpoint_id` ‚Äî must match the server
- `auth_profile` ‚Äî `shared-deployment` or `per-user`
- `admission_key` ‚Äî either the shared deployment key or the user's dedicated admission key
- `server_static_public_key` ‚Äî server public key
- `client_static_private_key` ‚Äî client's stable static private key
- `client_identity` is required for `auth_profile = "per-user"`
- `enable_s1_fallback` ‚Äî enables conservative UDP-to-stream fallback when `stream_server_addr` is present
- `allow_session_migration` ‚Äî enables authenticated path revalidation and standby promotion logic
- `standby_health_check_secs` ‚Äî override for standby probe cadence; `0` keeps the persona-selected sparse cadence
- `use_server_pushed_routes = true` is usually the easiest choice

## 4. Start the server

```bash
sudo ./target/release/apt-edge start --config /etc/adapt/server.toml
```

If you prefer boot-persistent service management on Linux, `apt-edge init` can now also install `/etc/systemd/system/apt-edge.service` for you when you answer `y` to the startup-service prompt or pass `--install-systemd-service`. If you skip that during init, you can later run `apt-edge utils install-systemd-service --config /etc/adapt/server.toml` to install or refresh the same unit.

Optional one-shot override:

- `--mode 0..100`

## 5. Generate or install a client bundle

For the supported client launch path, package the client as a single `.aptbundle` file and run `apt-client` against that bundle.

The normal operator flow is still:

```bash
sudo ./target/release/apt-edge add-client --config /etc/adapt/server.toml --name laptop --auth per-user
```

That produces a single-file bundle you can install as `/etc/adapt/client.aptbundle` on the client.
On first client run, `apt-client up` also creates a blank optional override file at `/etc/adapt/client.override.toml`.

## 6. Start the client

```bash
sudo ./target/release/apt-client up
```

If the bundle is installed at the default location `/etc/adapt/client.aptbundle`, the client stores persistent state at `/var/lib/adapt/client-state.toml`.
Use `/etc/adapt/client.override.toml` for client-local changes that should not require rebuilding the bundle.

Optional one-shot overrides:

- `--mode 0..100`
- `--carrier auto|d1|d2|s1`

To run a bundle from a non-default location:

```bash
sudo ./target/release/apt-client up --bundle /path/to/laptop.aptbundle
```

That direct-launch path creates a sidecar override file such as `/path/to/laptop.override.toml`.

Useful override fields include:

- `server_addr`
- `enable_d2_fallback`
- `d2_server_addr`
- `d2_server_certificate`
- `stream_server_addr`
- `mode`
- `preferred_carrier`
- `bind`
- `interface_name`
- `routes`
- `use_server_pushed_routes`
- `session_policy`
- `enable_s1_fallback`
- `allow_session_migration`
- `standby_health_check_secs`
- `keepalive_secs`
- `session_idle_timeout_secs`
- `handshake_timeout_secs`
- `handshake_retries`
- `udp_recv_buffer_bytes`
- `udp_send_buffer_bytes`
- `state_path`

## Config upgrade behavior

Server config files are best-effort normalized on load so newly added defaults appear automatically in older TOML files.

That is convenient for upgrades, but it also means:

- comments may be dropped
- formatting may be normalized
- key ordering may change after a newer binary loads and rewrites the file

## When to use this manual path

Use the manual path when you want:

- complete control over file layout
- pre-generated keys from another system
- hand-edited configs for automation or packaging
- to understand the exact raw config fields embedded inside generated client bundles and used by the guided CLI
