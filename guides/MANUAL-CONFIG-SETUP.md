# Manual config reference

This guide is for operators who want the lower-level raw config reference rather than the guided CLI flow.

The supported client distribution artifact is the single-file `.aptbundle` generated by `apt-edge add-client`.
The client example in this guide describes the **logical config embedded inside that bundle**.

If you just want the supported happy path, use `guides/DEPLOYMENT.md`.

## 1. Generate raw server keys

```bash
./target/release/apt-edge gen-keys --out-dir /etc/adapt
```

This produces:

- `/etc/adapt/shared-admission.key`
- `/etc/adapt/server-static-private.key`
- `/etc/adapt/server-static-public.key`
- `/etc/adapt/cookie.key`
- `/etc/adapt/ticket.key`

## 2. Generate a standalone client identity (advanced)

```bash
./target/release/apt-client gen-identity --out-dir /etc/adapt
```

This produces:

- `/etc/adapt/client-static-private.key`
- `/etc/adapt/client-static-public.key`

## 3. Prepare the config files manually

Use these templates:

- `guides/examples/server.toml`
- `guides/examples/client.toml`

### Server config fields

- `bind` — TCP/TLS listen address for the H2 API-sync service
- `public_endpoint` — public host:port clients should dial
- `authority` — HTTP authority / host name exposed by the public API-sync surface
- `certificate` — server certificate for the H2 surface; supports `file:/path`, inline PEM, or base64-encoded DER
- `private_key` — server private key; supports `file:/path` or inline PEM/key data
- `deployment_strength` — current H2 deployment posture (default generated configs use `self-contained`)
- `mode` — numeric runtime mode (`0` = speed, `50` = balanced, `100` = stealth)
- `endpoint_id` — deployment identifier
- the core 32-byte AdaPT keys support either inline hex or `file:/path`
- `tunnel_local_ipv4` + `tunnel_netmask` define the tunnel subnet
- `tunnel_local_ipv6` + `tunnel_ipv6_prefix_len` optionally enable an IPv6 tunnel subnet
- `enable_ipv4_forwarding` / `nat_ipv4` control Linux IPv4 forwarding and NAT
- `enable_ipv6_forwarding` / `nat_ipv6` control Linux IPv6 forwarding and NAT66
- `push_routes` usually contains `0.0.0.0/0` for full tunnel; add `::/0` too if you enable tunnel IPv6
- `push_dns` is applied automatically where supported by the client platform
- `allow_session_migration` keeps the authenticated session migration machinery enabled
- `[[peers]]` must include each authorized client public key and assigned tunnel IPv4, and can also assign `tunnel_ipv6`
- for per-user admission, each `[[peers]]` entry should also set:
  - `auth_profile = "per-user"`
  - `user_id = "..."` (or let it default to `name`)
  - `admission_key = "file:/path/to/user-specific-admission.key"`

### Client bundle config fields

- `server_addr` — public host:port the client connects to
- `authority` — HTTP authority / host name expected on the public API-sync surface
- `server_name` — TLS SNI / server-name override when needed; normally matches `authority`
- `server_roots` — optional CA roots bundle when you trust a CA instead of pinning a leaf certificate
- `server_certificate` — pinned H2 server certificate; supports `file:/path`, inline PEM, or base64-encoded DER
- `deployment_strength` — current H2 deployment posture
- `mode` — numeric runtime mode (`0` = speed, `50` = balanced, `100` = stealth)
- `endpoint_id` — must match the server
- `auth_profile` — `shared-deployment` or `per-user`
- `admission_key` — either the shared deployment key or the user's dedicated admission key
- `server_static_public_key` — server static public key
- `client_static_private_key` — client static private key
- `client_identity` is required for `auth_profile = "per-user"`
- `allow_session_migration` keeps the authenticated session migration machinery enabled
- `use_server_pushed_routes = true` is usually the easiest choice

## 4. Start the server

```bash
sudo ./target/release/apt-edge start --config /etc/adapt/server.toml
```

Optional one-shot override:

- `--mode 0..100`

If you prefer boot-persistent service management on Linux:

```bash
sudo ./target/release/apt-edge utils install-systemd-service --config /etc/adapt/server.toml
```

## 5. Package or install a client bundle

The supported client launch path is still a single `.aptbundle`.

The normal operator flow is:

```bash
sudo ./target/release/apt-edge add-client --config /etc/adapt/server.toml --name laptop --auth per-user
```

That produces a bundle you can install as `~/.adapt-tunnel/client.aptbundle` on the client.
On first use, `apt-client up` also creates a blank local override file at `~/.adapt-tunnel/client.override.toml`.

## 6. Start the client

```bash
mkdir -p ~/.adapt-tunnel
cp /path/to/laptop.aptbundle ~/.adapt-tunnel/client.aptbundle
sudo ./target/release/apt-client service install
./target/release/apt-client up
```

If the bundle is installed at the default location, the client stores persistent state at `~/.adapt-tunnel/client.state.toml`.

To run a bundle from a non-default location:

```bash
./target/release/apt-client up --bundle /path/to/laptop.aptbundle
```

That direct-launch path creates sidecar files such as `/path/to/laptop.override.toml`.

Useful override fields include:

- `server_addr`
- `authority`
- `server_name`
- `server_roots`
- `server_certificate`
- `mode`
- `deployment_strength`
- `bind`
- `interface_name`
- `routes`
- `use_server_pushed_routes`
- `session_policy`
- `allow_session_migration`
- `standby_health_check_secs`
- `keepalive_secs`
- `session_idle_timeout_secs`
- `handshake_timeout_secs`
- `handshake_retries`
- `udp_recv_buffer_bytes`
- `udp_send_buffer_bytes`
- `state_path`

## 7. Validate manually

Use `guides/MANUAL-TESTING.md`.

## Config upgrade behavior

Server and client config files are best-effort normalized on load so newly added defaults appear automatically in older TOML files.
That means comments/formatting may be normalized when a newer binary rewrites them.
